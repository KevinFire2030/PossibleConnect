# 테넌트(Tenant) 개념 설명

## 테넌트란?

**테넌트(Tenant)**는 "임차인" 또는 "사용자 그룹"을 의미합니다. 소프트웨어에서는 **데이터를 독립적으로 관리하는 논리적 단위**를 말합니다.

## 쉽게 이해하기

### 아파트 비유
- **아파트 건물** = 애플리케이션/시스템
- **각 세대** = 테넌트
- **각 세대의 방** = 테넌트의 데이터

각 세대는 독립적인 공간을 가지고 있으며, 다른 세대의 방에는 들어갈 수 없습니다.

## 소프트웨어에서의 테넌트

### 예시 1: 이메일 서비스
```
Gmail 서비스
├── 테넌트 A: user1@gmail.com
│   ├── 이메일 1
│   ├── 이메일 2
│   └── 연락처
├── 테넌트 B: user2@gmail.com
│   ├── 이메일 1
│   ├── 이메일 2
│   └── 연락처
└── 테넌트 C: user3@gmail.com
    ├── 이메일 1
    └── 연락처
```

각 사용자는 자신의 이메일과 데이터만 볼 수 있습니다.

### 예시 2: 클라우드 스토리지
```
Dropbox 서비스
├── 테넌트 A: 회사 A
│   ├── 문서 1
│   ├── 문서 2
│   └── 공유 폴더 A
├── 테넌트 B: 회사 B
│   ├── 문서 1
│   ├── 문서 2
│   └── 공유 폴더 B
└── 테넌트 C: 개인 사용자
    ├── 사진 1
    └── 사진 2
```

각 회사나 사용자는 자신의 데이터만 접근할 수 있습니다.

## 단일 테넌트 vs 멀티테넌트

### 단일 테넌트 (Single Tenant)

**하나의 테넌트만 존재하는 환경**

```
시스템
└── 테넌트 1 (동아리)
    ├── 동아리 소개
    ├── 회칙
    ├── 2026년 사업계획
    └── FAQ
```

**특징:**
- ✅ 데이터 격리가 필요 없음
- ✅ 모든 데이터를 함께 관리
- ✅ 검색 시 전체 데이터에서 검색
- ✅ 간단한 구조

**당신의 경우:**
- 동아리 소개, 회칙, 사업계획, FAQ를 하나의 컬렉션에 저장
- 모든 문서를 함께 검색
- → **단일 테넌트 환경**

### 멀티테넌트 (Multi-Tenant)

**여러 테넌트가 존재하는 환경**

```
시스템
├── 테넌트 A (동아리 A)
│   ├── 동아리 A 소개
│   ├── 동아리 A 회칙
│   └── 동아리 A FAQ
├── 테넌트 B (동아리 B)
│   ├── 동아리 B 소개
│   ├── 동아리 B 회칙
│   └── 동아리 B FAQ
└── 테넌트 C (동아리 C)
    ├── 동아리 C 소개
    └── 동아리 C 회칙
```

**특징:**
- ✅ 각 테넌트의 데이터가 격리됨
- ✅ 테넌트 A는 테넌트 B의 데이터를 볼 수 없음
- ✅ 데이터 보안 및 프라이버시 보장
- ✅ 복잡한 구조

**예시:**
- SaaS 서비스 (여러 회사가 같은 시스템 사용)
- 사용자별 개인 문서 관리
- 조직별 독립적인 데이터 관리

## Qdrant에서의 테넌트

### Global Search (단일 테넌트)
```
Collection: "possible"
├── 문서 1 (동아리 소개)
├── 문서 2 (회칙)
├── 문서 3 (사업계획)
└── 문서 4 (FAQ)

→ 모든 문서를 함께 검색
→ 데이터 격리 불필요
```

### Multitenancy (멀티테넌트)
```
Collection: "possible"
├── Tenant A (동아리 A)
│   ├── 문서 1
│   └── 문서 2
├── Tenant B (동아리 B)
│   ├── 문서 3
│   └── 문서 4
└── Tenant C (사용자 1)
    ├── 문서 5
    └── 문서 6

→ Tenant A가 검색하면 Tenant A의 문서만 검색
→ 다른 테넌트의 데이터는 접근 불가
```

## 실제 사용 사례

### 단일 테넌트가 적합한 경우
1. ✅ **동아리 문서 관리** (당신의 경우)
   - 하나의 동아리 문서를 관리
   - 모든 문서를 함께 검색

2. ✅ **회사 내부 지식 베이스**
   - 회사 전체 문서를 하나의 시스템에서 관리
   - 모든 직원이 모든 문서 검색 가능

3. ✅ **개인 문서 관리**
   - 개인의 모든 문서를 하나의 컬렉션에 저장
   - 전체 문서에서 검색

### 멀티테넌트가 적합한 경우
1. ✅ **SaaS 서비스**
   - 여러 회사가 같은 시스템 사용
   - 각 회사의 데이터는 격리

2. ✅ **사용자별 개인 문서**
   - 각 사용자가 자신의 문서만 관리
   - 다른 사용자의 문서는 접근 불가

3. ✅ **조직별 문서 관리**
   - 여러 조직이 같은 시스템 사용
   - 각 조직의 문서는 독립적으로 관리

## 요약

### 테넌트란?
- **데이터를 독립적으로 관리하는 논리적 단위**
- 아파트의 "세대"와 비슷한 개념

### 단일 테넌트
- **하나의 테넌트만 존재**
- 모든 데이터를 함께 관리
- 데이터 격리 불필요
- **당신의 경우에 적합** ✅

### 멀티테넌트
- **여러 테넌트가 존재**
- 각 테넌트의 데이터가 격리
- 데이터 보안 및 프라이버시 보장
- 여러 사용자/조직이 같은 시스템을 사용할 때 필요

## 결론

**당신의 경우:**
- 동아리 소개, 회칙, 사업계획, FAQ를 하나의 컬렉션에 저장
- 모든 문서를 함께 검색하여 RAG 수행
- → **단일 테넌트 환경**
- → **Global Search 선택** ✅

**멀티테넌트가 필요한 경우:**
- 여러 동아리가 같은 시스템을 사용하고
- 각 동아리의 문서를 격리해야 할 때
- → **Multitenancy 선택**
